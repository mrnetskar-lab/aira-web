======================================================================
PATCH GOAL
======================================================================

Introduce a stable character identity system now, before the project grows.

This patch separates:
- permanent internal identity
- legacy compatibility keys
- UI display names

That allows the team to:
- rename characters later
- keep autosave stable
- keep thread state stable
- keep relationship state stable
- keep future memory/logging stable
- support nicknames or altered names later

This patch should happen before:
- deeper persistence
- backend memory expansion
- dating app
- wider cast growth

======================================================================
CORE PRODUCT DECISION
======================================================================

Use three identity layers:

1. `id`
Permanent internal identity.
Never changes.

2. `key`
Short compatibility alias.
Can match old naming for now.
Used only while refactoring.

3. `displayName`
What the user sees.
Safe to rename later.

Example:
```json
{
  "id": "c001",
  "key": "lucy",
  "displayName": "Lucy"
}
```

======================================================================
1) NEW CHARACTER REGISTRY
======================================================================

CREATE:
`src/shared/characters.js`

ADD:

```js
export const CHARACTER_REGISTRY = {
  c001: {
    id: "c001",
    key: "lucy",
    displayName: "Lucy",
    colorKey: "Lucy"
  },
  c002: {
    id: "c002",
    key: "sam",
    displayName: "Sam",
    colorKey: "Sam"
  },
  c003: {
    id: "c003",
    key: "angie",
    displayName: "Angie",
    colorKey: "Angie"
  }
};

export function getCharacterById(id) {
  return CHARACTER_REGISTRY[id] || null;
}

export function getCharacterByKey(key) {
  const normalized = String(key || "").toLowerCase();
  return Object.values(CHARACTER_REGISTRY).find(
    (character) => character.key.toLowerCase() === normalized
  ) || null;
}

export function getCharacterDisplayName(idOrKey) {
  return (
    getCharacterById(idOrKey)?.displayName ||
    getCharacterByKey(idOrKey)?.displayName ||
    idOrKey ||
    "Unknown"
  );
}

export function normalizeCharacterId(idOrKey) {
  return (
    getCharacterById(idOrKey)?.id ||
    getCharacterByKey(idOrKey)?.id ||
    null
  );
}
```

RULE
New code should prefer `id`.
Old code may still pass `key` temporarily.

2) THREAD MODEL MIGRATION
======================================================================

TARGET
`client/app.js`

CURRENT PROBLEM
Messages threads and thread message maps still use:
- `lucy`
- `sam`
- `angie`

These should move to stable IDs.

CHANGE THREADS FROM:
```js
threads: [
  { id: "lucy", name: "Lucy", preview: "You disappeared.", unread: true },
  { id: "sam", name: "Sam", preview: "You there?", unread: false },
  { id: "angie", name: "Angie", preview: "Call me later", unread: true }
]
```

TO:
```js
threads: [
  { id: "c001", key: "lucy", displayName: "Lucy", preview: "You disappeared.", unread: true },
  { id: "c002", key: "sam", displayName: "Sam", preview: "You there?", unread: false },
  { id: "c003", key: "angie", displayName: "Angie", preview: "Call me later", unread: true }
]
```

CHANGE THREAD MESSAGES FROM:
```js
messages: {
  lucy: [...],
  sam: [...],
  angie: [...]
}
```

TO:
```js
messages: {
  c001: [...],
  c002: [...],
  c003: [...]
}
```

RULE
All thread lookups should use stable `id`.

UI should display:
```js
thread.displayName
```

3) ROOM CHAT RESPONSE CONTRACT
======================================================================

TARGET
AI response shape

CURRENT PROBLEM
Room chat currently treats `agent` as a display name.

NEW RULE
Responses should move toward:

```json
{
  "agentId": "c001",
  "agentKey": "lucy",
  "agentDisplayName": "Lucy",
  "spoken": "You came back.",
  "thought": "",
  "meta": {}
}
```

TRANSITION RULE
During migration, frontend should support both:
- old `agent`
- new `agentId` / `agentDisplayName`

FRONTEND NORMALIZATION RULE
In `normalizeResponses(payload)`:

```js
const agentId = entry.agentId || normalizeCharacterId(entry.agent) || null;
const agentDisplayName =
  entry.agentDisplayName ||
  getCharacterDisplayName(agentId || entry.agent) ||
  entry.agent ||
  "AIRA";
```

Then store:
```js
agentId,
agent: agentDisplayName,
```

This lets the UI keep working while the backend migrates.

function characterNameClass(idOrName) {

  if (colorKey === "Lucy") return "message__name--Lucy";

6) AUTOSAVE MIGRATION SAFETY
======================================================================

TARGET
`client/app.js`

CURRENT PROBLEM
Autosave may already contain old thread keys:
- lucy
- sam
- angie

ADD MIGRATION HELPER:

```js
function migrateLegacyMessagesApp(messagesApp) {
  if (!messagesApp) return messagesApp;

  const keyToId = {
    lucy: "c001",
    sam: "c002",
    angie: "c003"
  };

  const migratedThreads = Array.isArray(messagesApp.threads)
    ? messagesApp.threads.map((thread) => ({
        ...thread,
        id: keyToId[thread.id] || thread.id,
        key: thread.key || thread.id,
        displayName: thread.displayName || thread.name || getCharacterDisplayName(keyToId[thread.id] || thread.id)
      }))
    : [];

  const migratedMessages = {};
  const sourceMessages = messagesApp.messages || {};

  for (const [key, value] of Object.entries(sourceMessages)) {
    const nextKey = keyToId[key] || key;
    migratedMessages[nextKey] = value;
  }

  return {
    ...messagesApp,
    activeThread: keyToId[messagesApp.activeThread] || messagesApp.activeThread || null,
    threads: migratedThreads,
    messages: migratedMessages
  };
}
```

IN `restoreLocalState()`:
```js
if (data.messagesApp) {
  state.messagesApp = migrateLegacyMessagesApp(data.messagesApp);
}
```

This prevents old autosave data from breaking after the ID migration.
