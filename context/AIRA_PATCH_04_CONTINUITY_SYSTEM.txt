3) TIME MODEL
======================================================================

Use a fake internal turn clock.

RULE
Each run increments:
```js
state.continuity.turnClock += 1;
```

Map turnClock to a simple time block cycle:
- 0–3 => evening
- 4–6 => late_night
- 7–9 => sleep_hours
- 10–13 => morning
- 14–17 => day
- 18+ => wrap around

Simplest implementation:
```js
function getTimeBlock(turnClock) {
  const slot = turnClock % 18;
  if (slot <= 3) return 'evening';
  if (slot <= 6) return 'late_night';
  if (slot <= 9) return 'sleep_hours';
  if (slot <= 13) return 'morning';
  return 'day';
}
```

Why:
- enough to simulate rhythm
- enough to test sleep/unavailability
- cheap to implement

6) MINIMAL AVAILABILITY RULES
======================================================================

At each turn, for each character:

1. determine current time block
2. determine sleeping
3. determine busy chance
4. determine online flag
5. reset transient seen/replying markers unless actively used this turn

RULES
- if sleeping => online false, busy false, replying false
- if busy => online may still be true
- online does not guarantee reply
- seen can happen without reply
- replying should be rare and brief in this phase

SIMPLE RULES
```js
sleeping = profile.sleepDuring.includes(timeBlock)

busy =
  !sleeping &&
  Math.random() < profile.busyBias &&
  timeBlock !== 'late_night'

online =
  !sleeping &&
  (profile.prefers.includes(timeBlock) || Math.random() < 0.45)
```

7) PLAYER MESSAGE REACTION SIGNALS
======================================================================

When the player sends input:
- mark some characters as having potentially seen it
- decide whether one character should become `replying`
- optionally queue later follow-up if no immediate reply happens

Do NOT fake this for everyone every turn.

Rules:
- direct mention of a character increases seen + reply odds for that character
- stronger relationship increases seen + reply odds
- high resentment lowers reply odds
- high attachment may increase seen even if no reply
- high pride can cause seen-without-reply

DERIVED HELPER
```js
interestScore =
  trust * 0.25 +
  attraction * 0.25 +
  attachment * 0.25 +
  (100 - resentment) * 0.15 +
  (100 - pride) * 0.10
```

Use normalized values if your state is 0–1 instead of 0–100.

PHASE A RULE
Use only one or two continuity signals per turn max:
- seen
- replying
- short busy line
- delayed follow-up

Not all at once.

======================================================================
8) DELAYED FOLLOW-UP QUEUE
======================================================================

This is the most important part of the patch.

If a scene had emotional weight, but no immediate full reply happens, queue a later event.

QUEUE CONDITIONS
- high trust / attraction interaction
- mention of that character
- emotional keywords
- high tension scene
- apology scene
- soft vulnerability scene

QUEUE TYPES
- `soft_check_in`
- `busy_reply`
- `late_follow_up`

EXAMPLES
Lucy:
- "hey. i was thinking about what you said."

Sam:
- "busy. answer later."

Angie:
- "you disappeared on me :)"

Simple queue API:
```js
queueFollowUp({
  state,
  character: 'Lucy',
  type: 'soft_check_in',
  delayTurns: 2,
  reason: 'vulnerable_scene'
});
```

CONSUME RULE
At start of run, consume any ready queued events and attach them to result metadata.

Do NOT inject too many.
Hard cap:
- max 1 consumed follow-up event per run in Phase A

- busy messages feel robotic

======================================================================
14) CRITICAL DESIGN RULES
======================================================================

1. Continuity should create longing, not clutter.
2. Do not show all signals every turn.
3. Normal availability is as important as weird availability.
4. Small delays matter more than big explanations.
5. This patch is a test, not the final relationship sim.
